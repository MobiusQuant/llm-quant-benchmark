# T2 分析能力测评报告

## 1. 测评概述

本测评旨在评估主流大语言模型在量化交易场景下的**K线结构识别与分析能力**。

给定一段真实K线数据，要求模型识别其中的关键结构（高低点、SMC框架下的FVG/Order Block/BOS/CHoCH/EQH/EQL），将模型输出与标准答案对比，衡量其结构识别的准确性。

成本：约 $10.08（OpenRouter 11 模型）+ ~$4.25（Claude Fable 5 补测）+ ~$3.20（GLM 5.2 补测，智谱一方 API，3 轮，按官方价折算）

### 测评模型

共 13 个模型，覆盖 3 家海外厂商 + 6 家国产厂商：

| 分类 | 模型 |
|------|------|
| 御三家旗舰 | GPT-5.5、Claude Opus 4.8、Gemini 3.1 Pro |
| 御三家次旗舰 | Claude Opus 4.6、Claude Sonnet 4.6 |
| 国产第一梯队 | DeepSeek V4 Pro、Qwen3.7 Max、Kimi K2.6、GLM 5.1、MiniMax M3、MiMo V2.5 Pro |
| 后续补测 | Claude Fable 5（Anthropic 2026-06 最新旗舰）、GLM 5.2（智谱最新旗舰，一方 BigModel API） |

> **Claude Fable 5 补测条件说明**：经本地 Anthropic 中转代理接入；模型拒绝 `temperature` 参数（使用默认采样，其余模型为 0.0）；显式设置 `max_tokens=32000`（代理默认 8192 曾导致重型用例输出截断/为空，重跑后修复）；费用按官方定价（$10/$50 per 1M）折算。其成绩已经五步审计独立复核，见 T1 报告第 7 节与技术报告第 12 章。
>
> **GLM 5.2 补测条件说明**：经智谱官方 BigModel API（一方端点）接入，与 GLM 5.1（OpenRouter）渠道不同；`temperature=0`、`reasoning_effort=max`、`max_tokens=49152`、**3 轮独立采样**（其余模型 T2 为 1 轮）。费用按官方价（$1.4/$4.4 per 1M）折算，由新账号 20M 免费额度覆盖。详见 T1 报告与技术报告第 12 章。

### 测评规模

- 12 个测试用例 × 1 轮 × 12 个模型 + 12 × 3 轮（GLM 5.2）= **180 次 API 调用**
- 每个用例仅跑 1 轮（节省成本），分为给定义和不给定义两种模式
- 同步附带 JSON 格式规范性评分

## 2. 测评设计

### 2.1 题目设计

共 6 个子任务，按难度分层：

| 难度 | 子任务 | 说明 | 标准答案来源 |
|------|--------|------|-------------|
| 简单 | 最高点识别 | 找出 High 列最大值及其位置 | max(High) |
| 简单 | 最低点识别 | 找出 Low 列最小值及其位置 | min(Low) |
| 困难 | FVG 识别 | 识别公允价值缺口区域 | MobiusQuant API |
| 困难 | Order Block 识别 | 识别订单块区域 | MobiusQuant API |
| 困难 | BOS/CHoCH 识别 | 识别结构突破和性质转换事件 | MobiusQuant API |
| 困难 | EQH/EQL 识别 | 识别等值高低点（流动性堆积） | MobiusQuant API |

### 2.2 两种测试模式

每个子任务设计两种 prompt 模式：

- **给定义**：prompt 中提供 SMC 概念的完整定义
- **不给定义**：仅给出术语名称，模型需自行理解概念

### 2.3 数据来源

使用 BTC/USDT 1小时 K线真实行情数据（2024年1月初），共 200 根 K线。

### 2.4 评分标准

- **高低点**：row 编号 + price 精确匹配，满分 1.0
- **SMC 结构**：基于 timestamp 匹配计算 F1 score（precision × recall 的调和平均）
- **JSON 合规**：自动检查响应的 JSON 可解析性、纯净度、Schema 合规性、枚举值合规性

SMC 标准答案由 [MobiusQuant API](https://docs.mobiusquant.ai/zh/indicators/smc.html) 计算生成（swing_size=10, internal_size=5）。

## 4. 分维度分析

### 4.1 按子任务难度

| 子任务 | 全模型平均 | 最佳模型 | 最差模型 |
|--------|-----------|---------|---------|
| 最高点 | 95.9% | 11个模型 100% | Claude Sonnet 4.6 (50%) |
| 最低点 | 100% | 全部 100% | — |
| FVG | 18.8% | Claude Opus 4.8 (31.9%)，Fable 5 次之 (28.9%) | GLM 5.1 (7.4%) |
| Order Block | 8.3% | Qwen3.7 Max (19.2%) | 4个模型 (0%) |
| BOS/CHoCH | 10.9% | GPT-5.5 (30.3%)，Fable 5 次之 (26.3%) | 4个模型 (0%) |
| EQH/EQL | 40.0% | MiMo V2.5 Pro (75.0%) | GLM 5.1 (0%) |

> 平均值为 13 模型口径（含补测的 Claude Fable 5 与 GLM 5.2）。

**高低点 vs SMC 的鸿沟极大。** 高低点平均 98.0%，SMC 四项平均仅 19.5%。高低点本质是数值比较（找 max/min），而 SMC 需要理解多根 K 线之间的结构关系，是完全不同层次的认知能力。

**EQH/EQL 是 SMC 中得分最高的子任务（41%）。** 可能因为 EQH/EQL 的概念相对直观——"找两个差不多高/低的点"，而 OB 和 BOS/CHoCH 需要理解趋势、结构突破等更复杂的上下文。

**Order Block 和 BOS/CHoCH 几乎不可能。** 平均仅 8.1% 和 9.5%，多数模型得 0%。这两个概念需要模型同时理解趋势方向、关键位突破、和特定蜡烛的关系，远超当前 LLM 的结构推理能力。

## 7. 关键发现

### 发现一：T2 排名与 T1 发生大幅洗牌

| 模型 | T1 排名 | T2 排名 | 变化 |
|------|--------|--------|------|
| Claude Fable 5 | 1 | 4 | ↓3 |
| Claude Opus 4.8 | 2 | 6 | ↓4 |
| Claude Opus 4.6 | 3 | 9 | ↓6 |
| GLM 5.1 | 4 | 11 | ↓7 |
| Qwen3.7 Max | 12 | 2 | **↑10** |
| MiMo V2.5 Pro | 11 | 5 | **↑6** |
| Gemini 3.1 Pro | 8 | 1 | **↑7** |

计算能力（T1）和分析能力（T2）是完全不同的能力维度。T1 考验的是数值精度，T2 考验的是结构理解。在 T1 中表现优异的 Claude 系列和 GLM，在 T2 中明显下滑；而 T1 中垫底的 Qwen 和 MiMo，在 T2 中大幅上升。最有说服力的是 Claude Fable 5：T1 断层第一（83.3%，领先第二名 29.6 个百分点），T2 却只有 49.1%——**算力的代际飞跃没有带来结构理解的同步提升**。

### 发现二：SMC 结构识别是当前 LLM 的能力盲区

SMC 四项子任务的全模型平均得分：FVG 17.9%、OB 8.1%、BOS/CHoCH 9.5%、EQH/EQL 41.0%。Order Block 和 BOS/CHoCH 几乎全军覆没，4 个模型在这两项上得分为 0%。

这说明当前 LLM 对金融市场中"多根 K 线之间的结构性关系"缺乏有效的推理能力。这不是知识问题（给定义也没用），而是结构推理能力的根本局限。

### 发现三：给定义往往适得其反

全模型平均：给定义 43.8%，不给定义 46.9%。不给定义反而高出 3.1 个百分点。Gemini（-15.6%）、Claude Sonnet（-18.1%）、Qwen（-10.5%）在给定义时下降明显。

可能的原因：冗长的定义文本占据了模型的注意力，干扰了对 K 线数据本身的分析。模型更擅长利用自身的预训练知识直接分析，而不是在推理时临时消化一段新定义。

### 发现四：Claude 系列速度碾压但分数不占优——Fable 5 反向验证了这一点

Claude 旧三个模型平均响应时间 6-11 秒，其他模型 107-374 秒。但它们在 T2 中排名第 6/9/10，不如 Gemini 和 Qwen。Claude 旧系列的"快"来自于 completion tokens 极少（6-8K vs 其他模型 100-300K）——不展开推理直接给答案，在结构分析这种需要深度推理的任务上，"想得少"是劣势。

Claude Fable 5 恰好反向验证了这个结论：它把平均响应时间放慢到 62 秒、completion 烧到 63.5K token（Opus 4.8 的 11 倍），换来了 Claude 家族在 T2 的最好成绩（49.1%，第 4）——**"想得多"确实有用，但仍然没有突破 Gemini 的天花板**。

### 发现五：`eqhl_with_def` 是模型杀手

这个用例要求模型在 200 根 K 线中找出价格相近的高/低点对，本质是 O(n²) 的逐对比较。DeepSeek 花了 12 分钟，MiniMax 9 分钟，GPT-5.5 3.7 分钟，Kimi 和 GLM 直接超时。只有 Claude 系列能在 20 秒内完成——但它们靠的是"不深入分析就给答案"，准确率并不高。

### 发现六：MiMo V2.5 Pro 是 T2 的性价比黑马

$0.16 的成本拿到 47.0% 的得分（第5名），每分成本 $0.35，是所有模型中最低的。在 EQH/EQL 子任务上甚至拿到 75%（全场最高）。作为小米的端侧模型，这个表现超出预期。

### 发现七：思考型模型需要显式 max_tokens——默认 8192 是隐形杀手

Claude Fable 5 补测时，FVG/BOS 等重型用例最初全部返回空响应或截断 JSON：模型把代理默认的 8192 token 输出额度全部（或大部分）烧在内部思考上，正文无空间输出。显式设置 `max_tokens=32000` 后恢复正常——实测重型用例 completion 达 10K-16.6K，远超默认额度。

对量化 Agent 的工程启示：接入思考型模型时，`max_tokens` 必须按"思考预算 + 正文长度"预留，否则会出现难以排查的偶发空响应——**而且任务越重越容易触发，恰好挑中你最关心的场景**。
