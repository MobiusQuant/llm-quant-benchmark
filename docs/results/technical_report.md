# AI 量化 LLM 大横评 — 技术报告

## 摘要

本报告对 13 个主流大语言模型在量化交易场景下的能力进行了标准化横向评测。测评覆盖四大核心维度：技术指标计算（T1）、K 线结构分析（T2）、交易规则执行与约束遵循（T4+T7）、数据质量检测与长上下文保持（T5+T6），同时将 JSON 格式合规性作为贯穿所有维度的评估指标。

共执行 987 次 API 调用，使用 BTC/USDT 1 小时真实行情数据，所有评分均为自动化量化评估。

核心发现：(1) 补测的 Claude Fable 5（Anthropic 2026-06 旗舰）以 81.5% 成为首个综合突破 80% 的模型，并将 T1 计算上限从 53.7% 重写至 83.3%（给公式模式 27/27 全对），证明精确计算是可随代际进化解决的工程问题（但该模型**撰文时已下架、不可公开获取**，仅作能力天花板参照）；(2) **最近发布的国产之光 GLM 5.2 综合 72.7% 跃居第 3、反超 Claude Opus 4.8，且 T1 计算 61.1% 为全场第 2；在 Fable 5 下架后，它与 GPT-5.5 并列为实际可落地的最强**——计算代差正在被国产快速缩小；(3) 结构分析瓶颈依旧——连两代最新旗舰也只停在 39.9–49.1%，全场最高仅 52.4%；(4) LLM 在规则执行和异常检测上已接近可靠（均值 >92%），国产在执行类任务上已追平御三家，成本仅为其 1/5 至 1/15；(5) 在公开渠道（OpenRouter）可直接接入的模型中 GPT-5.5 综合最强（73.9%），而经一方 API 的 GLM 5.2（72.7%）已贴身紧追、成本仅其零头。

## 1. 引言

### 1.1 研究动机

在 AI 量化交易实践中，LLM 正被越来越多地引入策略研发流程——从技术指标计算、K 线形态分析，到交易信号生成和多 Agent 协作。然而，现有的 LLM 评测基准（如 MMLU、HumanEval）侧重通用能力，缺乏对量化交易垂直场景的针对性评估。

实际工作中发现的关键问题包括：
- 不同模型计算同一技术指标时，结果差异巨大，且不可预测
- 多 Agent 协作时，部分模型频繁输出非法 JSON，导致流水线中断
- 复杂交易规则的执行准确性未知，无法确定哪些决策可以信赖
- 缺少针对量化场景的横向对比数据，模型选型完全依赖直觉

本测评旨在用量化的方式回答一个核心问题：**在量化交易的具体任务中，哪个 LLM 最可靠、最高效？**

### 1.3 设计原则

1. **全部量化评分** — 不使用主观打分，所有维度均有确定性标准答案和自动化评分逻辑
2. **真实数据** — 使用 Binance BTC/USDT 1H 实盘行情，不使用合成数据
3. **统一接入** — 通过 OpenRouter API 统一调用所有模型，消除接入差异
4. **成本可控** — 记录每次调用的 token 消耗和费用，评估性价比

## 2. 方法论

### 2.1 数据来源

所有测评使用 BTC/USDT 1 小时 K 线真实行情数据，来自 Binance 交易所：

| 数据段 | 时间范围 | BTC 价格区间 | 用途 |
|--------|----------|:----------:|------|
| 2024-01 | 2024 年 1 月初 | ~42,000 – 49,000 | T1 / T2 / T4 / T5 |
| 2024-06 | 2024 年 6 月初 | ~67,000 – 72,000 | T1 |
| 2025-01 | 2025 年 1 月初 | ~93,000 – 99,000 | T1 |

每段数据包含 500 根 K 线，字段：timestamp、open、high、low、close、volume。

### 2.2 标准答案生成

| 维度 | 标准答案来源 | 算法/接口 |
|------|------------|----------|
| T1 | 自实现参考代码 | SMA（算术平均）、RSI（Wilder 平滑法）、MACD（SMA 初始化 EMA） |
| T2 高低点 | 数据列极值 | max(high) / min(low) |
| T2 SMC | MobiusQuant API | FVG / Order Block / BOS/CHoCH / EQH/EQL（swing_size=10, internal_size=5） |
| T4 | 自实现参考代码 | 条件组合逻辑判断 + MobiusQuant API 预算指标值 |
| T5 | 预埋注入 | 在真实数据中的确定性位置注入 5 种已知异常 |

所有参考实现代码开源于 `tests/*/reference.py`。

### 2.3 评分体系

| 评分方法 | 适用场景 | 算法 |
|---------|---------|------|
| `numeric_tolerance` | SMA / RSI 数值 | 相对误差 ≤ 阈值则通过 |
| `macd_tolerance` | MACD 三值 | DIF/DEA/MACD 分别评分，总分 = 通过项/3 |
| `highlow_match` | 高低点 | row + price 精确匹配 |
| `smc_f1` | SMC 结构 | 基于 timestamp 匹配的 F1 score |
| `rule_signal` | 交易信号 | 每根 K 线信号类型匹配 |
| `checklist` | 约束遵守 | 逐条 check，通过率 |
| `anomaly_f1` | 异常检测 | row + type 匹配的 F1 score |
| `json_compliance` | JSON 合规 | 可解析性 × 纯净度 × Schema × 枚举 |

**容差设定：**
- SMA：相对误差 ≤ 0.1%
- RSI：绝对误差 ≤ 1.0
- MACD：每项容差 = max(|标准答案| × 1%, 1.0)

### 2.4 测试模式

每个维度均设计了两种 prompt 模式，以区分"知识记忆"和"执行能力"：

| 维度 | 模式 A | 模式 B | 目的 |
|------|--------|--------|------|
| T1 | 给公式 | 不给公式 | 有无计算步骤说明是否影响准确率 |
| T2 | 给定义 | 不给定义 | 有无 SMC 概念定义是否帮助识别 |
| T4 | 给规则 | 不给规则 | 有无完整规则描述是否影响信号准确率 |

### 2.5 技术实现

- **调用方式**：OpenRouter API（统一入口），httpx 异步客户端；支持通过独立 config 切换 base_url/api_key 接入 OpenAI 兼容的本地代理（Fable 5 补测即此方式）
- **请求参数**：temperature 默认 0.0，可配置为 null（不发送，适配拒绝该参数的模型）；max_tokens 可配置（思考型模型需显式给足，否则思考耗尽输出额度会导致空响应）
- **并发控制**：semaphore 限制并发数（OpenRouter 6；本地代理 2，避免触发其并发限制）
- **重试策略**：429/5xx/缺 content 自动重试
- **超时设置**：300 秒
- **断点续跑**：每完成一个 API 调用立即写入磁盘，支持 `--resume` 恢复；失败记录移入 `results/_failed_rounds/` 归档后可精准补跑
- **结果归档**：`results/{run_id}/{dim}/{model}/{case}/round_{n}.json`

## 3. T1：技术指标计算能力

### 3.1 测评设计

选取 3 个代表性技术指标，按计算复杂度分三层：

| 难度 | 指标 | 计算特点 | 参数 |
|------|------|---------|------|
| 简单 | SMA | 纯算术平均 | SMA(10) |
| 中等 | RSI | 多步递推 + Wilder 平滑 | RSI(14) |
| 困难 | MACD | 多层 EMA 嵌套递推 | MACD(12,26,9) |

共 18 个用例（3 指标 × 2 模式 × 3 数据段），每用例跑 3 轮，合计 594 次 API 调用。

### 3.2 总体结果

| 排名 | 模型 | 综合 | SMA | RSI | MACD |
|:----:|------|:----:|:---:|:---:|:----:|
| 1 | **Claude Fable 5** | **83.3%** | **100%** | **100%** | **50.0%** |
| 2 | **GLM 5.2** | **61.1%** | 72.2% | 72.2% | 38.9% |
| 3 | Claude Opus 4.8 | **53.7%** | 5.6% | 100% | 11.1% |
| 4 | Claude Opus 4.6 | **51.8%** | 11.1% | 100% | 16.7% |
| 5 | GLM 5.1 | **48.1%** | 66.7% | 66.7% | 5.6% |
| 6 | GPT-5.5 | **46.3%** | 27.8% | 83.3% | 5.6% |
| 7 | Claude Sonnet 4.6 | **44.4%** | 16.7% | 100% | 0.0% |
| 8 | DeepSeek V4 Pro | **40.7%** | 50.0% | 66.7% | 5.6% |
| 9 | Gemini 3.1 Pro | **38.9%** | 66.7% | 50.0% | 0.0% |
| 10 | MiniMax M2.7 | **27.8%** | 16.7% | 38.9% | 0.0% |
| 11 | Kimi K2.6 | **24.1%** | 22.2% | 27.8% | 0.0% |
| 12 | MiMo V2.5 Pro | **16.7%** | 11.1% | 22.2% | 0.0% |
| 13 | Qwen3.7 Max | **3.7%** | 5.6% | 5.6% | 0.0% |

### 3.3 关键发现

**Claude Fable 5 重写了计算上限。** 补测的 Fable 5 综合 83.3%，比原冠军高 29.6 个百分点：SMA/RSI 双满分（各 18/18），给公式模式 27/27 全对（含 MACD 40 步递推），三个数据段得分完全一致（零波动）。代价是 completion token 巨大（137K，旧 Claude 的数倍）——通过显式长推理换取精度。

**GLM 5.2 是计算榜眼。** 国产新旗舰综合 61.1%（全场第 2，仅次于 Fable 5），SMA/RSI 各 72.2%，同样靠满档思考换取精度（`reasoning_effort=max`，单条平均约 11K completion token、3–4 分钟）。但 MACD 仍是短板（38.9%）：不给公式近乎全错、给公式才到 66.7%——与 Fable 5 一样卡在 EMA 初始化约定。它经智谱一方 API、3 轮独立采样，各轮真实波动（如 `rounds=[0 1 1]`，非缓存），可复现性优于经本地代理、三轮高度雷同的 Fable 5。

**反直觉：SMA 通过率低于 RSI。** SMA 仅需取 10 个数求平均（概念最简单），但全模型平均通过率仅 33.4%，低于 RSI 的 62.1%。原因：SMA 要从 30 根 K 线中定位"最后 10 根"的收盘价并准确相加——LLM 在"数数"和"大数求和"上频繁出错。RSI 计算步骤虽多，但涉及的数字较小（价格变动幅度），计算难度反而更低。

**MACD 全军覆没——Fable 5 除外。** 旧 11 模型平均通过率仅 4%，7 个为 0%。Fable 5 给公式时 9/9 全对；但不给公式时 0/9，失败模式是系统性偏差（DIF 算出 565.80 vs 标准答案 654.06）而非算术噪声——它默认采用了不同的 EMA 初始化约定。**执行已可靠，方法约定必须显式指定。**

**给公式不一定有帮助（对旧模型）。** 多数模型给公式时表现更好（GPT-5.5 +18.5%），但 GLM 5.1 不给公式时反而高出 18.6%——冗长的公式文本干扰了其执行流程。Fable 5 是给公式收益最大的模型（+33.3%：100% vs 66.7%）。

**计算结果不可复现（旧模型）。** 对 Claude Opus 4.8 进行 RSI 复测（初测 100%），9 次调用仅 1 次通过（11.1%）。同一模型在不同时间调用计算结果波动较大，初测的高通过率存在运气成分。Fable 5 的 54 次调用三轮间高度一致，但同样未做跨时间复测。

### 3.4 费用

T1 维度总花费约 $6.00（11 模型实测）+ $7.96（Fable 5 按官方定价折算）。Claude Opus 4.8 以 $1.25 拿到旧模型最高分（53.7%），GLM 5.1 以 $0.23 拿到 48.1%，每分成本仅为 Claude 的 1/5。Fable 5 的折算费用（$7.96）全场最高，但得分断层第一。

## 4. T2：K 线结构分析能力

### 4.1 测评设计

6 个子任务，按难度分层：

| 难度 | 子任务 | 标准答案来源 |
|------|--------|------------|
| 简单 | 最高点识别 | max(High) |
| 简单 | 最低点识别 | min(Low) |
| 困难 | FVG（公允价值缺口） | MobiusQuant API |
| 困难 | Order Block（订单块） | MobiusQuant API |
| 困难 | BOS/CHoCH（结构突破/性质转换） | MobiusQuant API |
| 困难 | EQH/EQL（等值高低点） | MobiusQuant API |

使用 200 根 K 线数据（2024-01），共 12 个用例（6 子任务 × 2 模式），180 次 API 调用（12 模型 × 1 轮 + GLM 5.2 × 3 轮）。

| 3 | GPT-5.5 | **49.4%** | 100% | 20.4% | 16.1% | 30.3% | 29.5% |
| 4 | **Claude Fable 5** | **49.1%** | 100% | 28.9% | 10.5% | 26.3% | 28.7% |

| 6 | Claude Opus 4.8 | **46.2%** | 100% | 31.9% | 5.0% | 3.8% | 36.5% |
| 7 | MiniMax M3 | **45.5%** | 100% | 12.2% | 5.3% | 8.0% | 47.2% |

### 4.3 关键发现

**高低点 vs SMC 的鸿沟极大。** 高低点平均 98.0%，SMC 四项平均仅 19.5%。高低点是简单的数值比较，而 SMC 需要理解多根 K 线之间的结构关系——两者是完全不同层次的认知能力。

**Order Block 和 BOS/CHoCH 几乎不可能。** 平均仅 8.3% 和 10.9%，多数模型得 0%。当前 LLM 对金融市场"多根 K 线之间的结构性关系"缺乏有效推理能力。

**结构分析天花板未被新一代突破。** T1 断层第一的 Claude Fable 5 在 T2 只有 49.1%（第 4）——它烧掉 63.5K completion token 深度思考（旧 Claude 的 11 倍、平均 62 秒/用例），换来 Claude 家族最好的 T2 成绩，但仍未越过 Gemini 的 52.4%。**算力的代际飞跃没有带来结构理解的同步提升。**

**给定义往往适得其反。** 旧 11 模型平均：给定义 43.8%，不给定义 46.9%。Gemini（-15.6%）、Claude Sonnet（-18.1%）在给定义时下降明显。模型更擅长利用预训练知识直接分析，冗长的定义文本反而占据注意力。

**T1 → T2 排名大幅洗牌。** Claude Opus 4.8 从 T1 第 2 跌至 T2 第 6，Qwen3.7 Max 从 T1 末位升至 T2 第 2。计算能力和分析能力是完全不同的能力维度。

**响应速度差异悬殊。** Claude 旧系列平均 6-11 秒完成，其他模型动辄 2-6 分钟；思考型的 Fable 5 平均 62 秒。Kimi K2.6 和 GLM 5.1 在 EQH/EQL 子任务上超时（>30 分钟）。

**思考型模型需要显式 max_tokens。** Fable 5 补测时，FVG/BOS 等重型用例在代理默认 8192 输出额度下全部空响应/截断——思考耗尽了输出额度。显式设置 32000 后正常（实测重型用例 completion 达 10K-16.6K）。任务越重越容易触发，工程接入时必须预留"思考预算 + 正文长度"。

### 4.4 费用

T2 维度总花费约 $10.08（11 模型实测）+ $4.25（Fable 5 按官方定价折算）。MiMo V2.5 Pro 以 $0.16 拿到第 5 名（47.0%），Gemini 3.1 Pro 虽然第 1 但花费 $3.01——生成了 222K completion tokens，是 Claude 旧系列（6-8K）的 30 倍。

## 5. T4+T7：规则执行与约束遵循

### 5.1 测评设计

模拟量化交易 Skill 调用场景，一次调用同时评估三个维度：

| 题目 | 规则类型 | 约束数量 | 说明 |
|------|---------|:-------:|------|
| Q3 | OR + AND 组合 | 4 | 4 个条件组判断 BUY/SELL/HOLD，SELL 优先 |
| Q4 | 嵌套条件 + 优先级 | 5 | 先判趋势方向，再基于趋势生成信号 |

每题 2 种模式（给/不给完整规则），共 4 个用例，44 次 API 调用。

## 10. 结论与建议

### 10.1 七大核心结论

**结论一：Claude Fable 5 综合登顶（81.5%），首个突破 80% 的模型。** T1 断层第一、T5 满分、全维度前五。其反常的计算成绩经五步审计独立复核（官方端点逐字复现，见第 12 章），确认为模型真实能力。它经本地代理补测（折算 $12.85 全场最贵），测试条件与主测不完全一致（见第 11 章）。

**结论二：计算瓶颈已被新一代攻破，结构分析瓶颈依旧。** Fable 5 把 T1 从"最高 53.7%"拉到 83.3%（给公式 27/27 全对，含 MACD 40 步递推），证明精确计算是可随代际进化解决的工程问题；但 T2 连它也停在 49.1%，多 K 线结构推理仍是全行业未破的认知瓶颈。

**结论三：GPT-5.5 是公开渠道（OpenRouter）可直接接入的最强模型。** 73.9% 综合第 2，T4/T5 双满分，JSON 全程 100%。但 $4.08 的总费用是 DeepSeek 的 9 倍、GLM 的 4 倍。

**结论四：国产模型在执行类任务上已达到御三家水平。** T4/T5 两个维度上，国产模型平均得分（92.8%）与御三家（92.4%）几乎相同（主测 11 模型口径）。但在 T1/T2 上，国产模型平均 33.5%，御三家 46.7%——补测的 Fable 5 进一步把 T1 代差拉大到 35 个百分点。

**结论五："没有全能冠军"首次松动，但按任务选型仍是最优解。** Fable 5 是第一个全维度进前五的模型，让"单模型兜底"变得可行；但它的维度极差仍有 50.9%，排名照样洗牌（T1 第 1 → T4 并列第 5）。

**结论六：JSON 合规性决定了 Agent 协作的可靠性。** GPT-5.5 和 DeepSeek V4 Pro 全程 100%，其输出可直接 `json.loads()` 解析。综合冠军 Fable 5 只有 85.9%（Claude 家族代码块习惯跨代未改），GLM 5.1 仅 73.6%——得分高不等于工程友好。

**结论七：GLM 5.1 仍是性价比最优选择。** $0.96 的总费用拿到 69.7%（第 4 名），每分成本是 GPT-5.5 的 1/4、Fable 5 的 1/11。

## 11. 局限性与改进方向

### 11.1 当前局限

1. **T1 数据部分丢失** — 因文件落盘 bug，T1 的详细 per-call 数据丢失，仅保留 summary 级别分数。后续测评已修复此问题。

2. **T2/T4/T5 仅 1 轮** — 为节省成本，T2 及之后的测评仅跑 1 轮，无法评估稳定性。T1 的 3 轮数据和后续复测显示部分模型稳定性差异显著。

3. **T5 异常过于明显** — 注入的异常（价格骤降 90%、负成交量）容易识别，导致 5 个模型满分。区分度不足。

4. **MiniMax 版本不一致** — T1 使用 M2.7，T2+ 使用 M3。合并排名可能不完全反映单一版本水平。

5. **单一数据源** — 仅使用 BTC/USDT 1H K 线，未覆盖其他币种、时间周期和市场状态。

6. **缺少多轮对话测评** — 均为单轮问答，未测试多轮 Agent 交互中的上下文保持和任务连贯性。

7. **Claude Fable 5 补测条件不完全一致** — ① 经本地 Anthropic 中转代理接入（账号池架构），其余模型走 OpenRouter；② 该模型拒绝 `temperature` 参数，使用默认采样（其余模型为 0.0）；③ 显式设置 `max_tokens=32000`（代理默认 8192 曾导致重型用例空响应/截断，归档后重跑修复，原始失败记录保存在 `results/_failed_rounds/`）；④ 代理每次调用注入约 515 token 开销（实测），token 统计偏高约 3%；⑤ 代理疑似响应缓存——三轮响应逐字节相同，**其三轮通过率不构成独立稳定性证据**；⑥ 费用按官方定价（$10/$50 per 1M）折算。针对以上风险已执行五步审计（见第 12 章）：评分管线零偏差、官方端点 4/4 复现，**能力结论可信，渠道侧问题仅影响 token 统计与轮间独立性**。

### 11.2 改进方向

1. **增加微妙异常** — 设计 2-3% 偏差的价格异常、时间戳微偏移等，提高 T5 区分度。

2. **增加多轮对话维度** — 测试模型在多轮 Agent 交互中的状态保持能力，模拟真实的策略研讨场景。

3. **扩展数据源** — 覆盖多币种（ETH、SOL）、多时间周期（5m、4h、1d）、极端行情（急涨急跌）。

4. **增加代码生成维度** — 虽然本次有意排除了代码生成（因已有大量 benchmark），但量化策略编写是重要的实际需求。

5. **增加轮次** — 所有维度统一跑 3-5 轮，评估稳定性和置信区间。

6. **自动化 CI** — 建立持续评测流水线，在新模型发布时自动跑分。

## 12. 结果可信度审计与诊断方案

Claude Fable 5 的 T1 成绩（83.3%）超出旧模型上限近 30 个百分点，且经由非标准渠道（本地账号池代理）补测——**结果好到值得怀疑**。我们由此设计并执行了一套五步诊断方案，对其全部成绩做独立审计。方案本身可复用于任何"成绩反常"的测评场景，故在此完整记录。

### 12.1 诊断方案（五步法）

| 步骤 | 检查内容 | 排除的风险 | 判定标准 |
|:----:|---------|-----------|---------|
| 1 | **管线完整性**：用当前评分器对存档原始响应离线重评分；存档期望答案 vs 用例源文件逐字比对 | 评分器 bug、答案篡改/漂移 | 重评分与存档分数零偏差；期望答案与源文件一致 |
| 2 | **误差指纹**：统计模型答案与标准答案的误差分布 | 区分"确定性计算"与"采样式心算" | 心算误差应有散布（参照同类模型）；批量零误差 = 计算指纹 |
| 3 | **跨轮一致性**：多轮原始响应是否逐字节相同 | 代理缓存、轮间非独立 | 采样温度 > 0 时长输出多轮全同 = 缓存嫌疑，轮次作废为单样本 |
| 4 | **高精度取证**：用 Decimal 重算精确值，比对模型答案匹配哪种舍入约定 | 参考实现自身的浮点/舍入误差 | 模型若匹配"精确值的规范舍入"，说明其计算精度达到了小数点后第 3 位 |
| 5 | **渠道对照**：用官方端点复跑判别性最强的用例 + 工具探针 | 代理后端工具作弊、注入干扰 | 官方裸模型能否复现；SHA-256 探针验证有无代码执行 |

**工具探针设计要点**：要求计算一个新颖字符串（不可能在训练数据中出现）的 SHA-256。纯语言模型在数学上不可能通过思维链完成 64 轮位运算的精确传播，**答对 = 后端有代码执行，答错/拒绝 = 裸模型**。这是区分"模型能力"与"Agent 能力"的最锐利探针。

**用例选择要点**：优先选有"舍入指纹"的用例做渠道对照。本次 SMA 2024-06 的精确均值恰为 67783.655（半进位 → .66，浮点 round → .65），模型答 .66 还是 .65 直接暴露其计算路径——这类天然判别点比随机抽查的信息量大得多。

### 12.2 Fable 5 审计结果

**Step 1 通过——脚本无错。** 73/73 个存档响应重评分零偏差；标准答案与 YAML 无差异。

**Step 2 发现计算指纹。** SMA/RSI 36 个答案误差中位数 0.0000（最大 0.01，30 个全零）；MACD 给公式 9 个全部 ±0.01；MACD 不给公式的"错误答案"内部精确自洽（MACD 柱 = 2×(DIF−DEA) 严格成立）。对照：Claude Opus 4.8 复测 RSI 的误差散布为 0.40-5.44。

**Step 3 发现缓存嫌疑。** 18 个用例中 16 个三轮响应逐字节相同 → 代理疑似按请求缓存，三轮通过率不构成独立稳定性证据。

**Step 4 模型比参考实现更精确。** SMA 2024-06 精确均值 67783.655 整；模型答 67783.66（规范半进位），Python 参考实现因浮点表示给出 67783.65。模型把十个 5 位数的和精确算到了第三位小数。

**Step 5 官方端点 4/4 复现。**

| 探针 | 官方 anthropic/claude-fable-5 | 代理版 | 标准答案 |
|------|------------------------------|--------|---------|
| SHA-256 工具探针 | 承认无法心算 ✓（无工具基线成立） | —（账号池耗尽未测） | — |
| SMA 2024-06 | **67783.66**（reasoning 仅 36 token） | 67783.66 | 67783.65（精确值 .655） |
| RSI 2024-01 | **76.89** | 76.89 | 76.89 |
| MACD 给公式 2024-01 | **654.06 / 779.45 / −250.76** | 同 | 完全一致 |

官方端点同时确认：真实定价 $10/$50 per 1M（Opus 4.8 的 2 倍）；代理注入开销 515 token/次（1,778 vs 1,263）。探针总花费约 $0.57。

### 12.3 审计结论

1. **测评脚本与评分体系无错误**（Step 1）
2. **Fable 5 的计算能力是模型真实内生能力**（Step 2/4/5）——跨渠道、跨账号、隔日复现，且 reasoning token 极少仍精确命中，说明高精度算术已内化，不依赖外部工具
3. **代理渠道存在两个真实问题，不影响能力结论**：注入 515 token/次（token 统计偏高约 3%）；疑似响应缓存（轮间独立性作废，确定性由官方抽查另行证实）
4. **方法论启示**：当某个模型成绩反常时，"误差指纹 + 舍入取证 + 渠道对照"三件套能在 1 美元以内完成定责——区分脚本 bug、渠道作弊、模型真实能力三种情形

## 附录 B：评分方法详细说明

### B.1 numeric_tolerance（SMA / RSI）

```
relative_error = |model_answer - reference| / |reference|
pass = relative_error <= tolerance
```

SMA tolerance = 0.1%，RSI tolerance = 1.0（绝对值）。

### B.2 macd_tolerance（MACD）

DIF、DEA、MACD 三个值分别评分：

```
per_value_tolerance = max(|reference| * 0.01, 1.0)
per_value_pass = |model - reference| <= per_value_tolerance
total_score = count(pass) / 3
```

### B.3 highlow_match（高低点）

```
score = 1.0 if (row_match AND price_match) else 0.0
```

row 和 price 都必须精确匹配。

### B.4 smc_f1（SMC 结构）

基于 timestamp 匹配计算 F1 score：

```
TP = model ∩ reference (by timestamp)
FP = model - reference
FN = reference - model
F1 = 2 * TP / (2 * TP + FP + FN)
```

### B.5 anomaly_f1（异常检测）

基于 row + type 联合匹配的 F1 score：

```
match = (row_match AND type_match)
F1 = 2 * Precision * Recall / (Precision + Recall)
```

### B.6 rule_signal（交易信号）

逐根 K 线比较信号类型：

```
per_candle_score = 1.0 if model_signal == reference_signal else 0.0
total_score = sum(per_candle_scores) / num_candles
```

### B.7 checklist（约束遵守）

逐条检查约束是否被遵守：

```
per_constraint_score = 1.0 if satisfied else 0.0
total_score = sum(scores) / num_constraints
```

### B.8 json_compliance（JSON 合规）

多项检查的加权平均：

- direct_parseable：输出能否直接 `json.loads()` 解析（0/1）
- no_code_fence：是否无 markdown 代码块包裹（0/1）
- no_extra_text：是否无 JSON 外的多余文字（0/1）
- extractable：即使有包裹，能否提取出合法 JSON（0/1）
- correct_type：根 JSON 类型是否正确（object/array）（0/1）
- field_presence：必填字段是否齐全（占比）
- field_type：字段类型是否正确（占比）
- enum_valid：枚举字段值是否合法（占比）
